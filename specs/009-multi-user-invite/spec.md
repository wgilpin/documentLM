# Feature Specification: Multi-User Access with Invite Codes and Document Privacy

**Feature Branch**: `009-multi-user-invite`
**Created**: 2026-03-17
**Status**: Draft
**Input**: User description: "make the app multi user. Total isolation - no one else can see your documents or see your sources / chunks in the db. Access is by invite code. A cli script to generate the codes - no walkin signups. For a given user, all their chunks / sources are shared, so an agent in one chat might find other chunks you added. A doc can be marked as Private - this means its sources and chunks are flagged as private and wont show up even in your other docs"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Invite-Only Registration and Login (Priority: P1)

A new user receives an invite code from an administrator. They visit the app, enter their invite code along with a chosen username and password, and gain access. Existing users log in with their credentials. All users who have not been invited are refused access — there is no self-service sign-up path.

**Why this priority**: Without access control, no other multi-user feature has meaning. This is the gate that all other stories depend on.

**Independent Test**: Can be fully tested by: (1) running the CLI tool to create an invite code, (2) registering with that code, (3) confirming a second registration attempt with the same code fails, and (4) confirming an unregistered visitor cannot access any content.

**Acceptance Scenarios**:

1. **Given** a valid, unused invite code, **When** a new user submits registration with that code, an email address, and a password, **Then** their account is created and they are logged in.
2. **Given** an invite code that has already been used, **When** someone attempts to register with it, **Then** registration is rejected with a clear error message.
3. **Given** an unauthenticated visitor, **When** they navigate to any page in the app, **Then** they are redirected to the login page and cannot access any content.
4. **Given** a registered user, **When** they log in with correct credentials, **Then** they access their own workspace.
5. **Given** a registered user, **When** they log in with incorrect credentials, **Then** access is denied.

---

### User Story 2 - Complete Data Isolation Between Users (Priority: P1)

Every user's documents, sources, and chunks are completely invisible to every other user. Accessing another user's resource by guessing its URL or ID returns a not-found or forbidden response. No cross-user data leaks via search, listings, or agent responses.

**Why this priority**: Without isolation, multi-user support creates a serious privacy and security failure. This has equal priority to P1 because it must be enforced from the moment any user data exists.

**Independent Test**: Can be fully tested by creating two users (A and B), adding a document to each, and confirming user B's session returns zero results for user A's documents, sources, and chunks — including via direct URL access.

**Acceptance Scenarios**:

1. **Given** users A and B each have documents, **When** user B views their document list, **Then** only user B's documents appear.
2. **Given** user A has a document with a known ID, **When** user B attempts to access it directly by URL, **Then** they receive a not-found or forbidden response.
3. **Given** user A has sources and chunks, **When** an agent running in user B's session searches for content, **Then** none of user A's sources or chunks are returned.
4. **Given** user A has sources and chunks, **When** user B views their source list, **Then** only user B's sources are visible.

---

### User Story 3 - Admin CLI for Invite Code Generation (Priority: P2)

An administrator runs a command-line script to generate one or more invite codes. The script outputs the code(s) so the admin can share them out-of-band (e.g., by email). No web UI is provided for this; it is a server-side administrative operation only.

**Why this priority**: This is required for any new user to gain access, but it is an admin-only operation and does not affect existing users' day-to-day experience.

**Independent Test**: Can be fully tested by running the CLI script with various arguments (single code, batch of N codes) and confirming each generated code can be used exactly once for registration.

**Acceptance Scenarios**:

1. **Given** the CLI script is run with no arguments, **When** executed, **Then** it generates and prints one valid invite code.
2. **Given** the CLI script is run with a count argument (e.g., `--count 5`), **When** executed, **Then** it generates and prints five unique invite codes.
3. **Given** a code generated by the CLI, **When** used to register, **Then** the account is created successfully.
4. **Given** the same code is used a second time, **When** registration is attempted, **Then** it is rejected.
5. **Given** a code is generated, **When** it has not been used, **Then** it remains valid indefinitely (no expiry by default).

---

### User Story 4 - Shared Knowledge Pool Within a User (Priority: P2)

When an agent assists a user inside any document, it can search across all non-private sources and chunks belonging to that user — regardless of which document originally added them. This means if a user added a source to Document A, an agent working on Document B can still surface relevant content from that source.

**Why this priority**: This is the key knowledge-sharing UX benefit within a single user's workspace, enabling a personal knowledge base that spans all their work.

**Independent Test**: Can be fully tested by: adding a source to Document A, then opening Document B and asking the agent a question whose answer is only found in Document A's source — the agent should return the relevant content.

**Acceptance Scenarios**:

1. **Given** a user has source S added to document A (not marked Private), **When** an agent in document B searches for content related to S, **Then** chunks from S appear in the agent's results.
2. **Given** a user has sources in three different documents (all non-private), **When** an agent in any one of those documents searches, **Then** it can find content from all three.
3. **Given** two users each have a source on the same topic, **When** user A's agent searches, **Then** it only finds user A's content, never user B's.

---

### User Story 5 - Private Document Mode (Priority: P3)

A user can mark any of their documents as Private. When a document is Private, its associated sources and chunks are invisible to agents and search in all other documents belonging to that user. The content is only accessible within the Private document itself. The user can toggle privacy on or off at any time.

**Why this priority**: This is an opt-in refinement to Story 4. The shared-knowledge behaviour (Story 4) is the default; Privacy is the exception a user can apply to sensitive documents.

**Independent Test**: Can be fully tested by: marking Document A as Private, then confirming an agent in Document B cannot find content from Document A's sources, then unmarking Document A and confirming the agent can now find it.

**Acceptance Scenarios**:

1. **Given** document A is marked Private and document B is not, **When** an agent in document B searches for content, **Then** no sources or chunks from document A appear.
2. **Given** document A is marked Private, **When** an agent working within document A itself searches, **Then** it can still access document A's own sources and chunks.
3. **Given** document A is marked Private, **When** the user removes the Private flag, **Then** document A's sources and chunks immediately become visible to agents in other documents.
4. **Given** a document is not Private by default, **When** the user marks it Private, **Then** all existing sources and chunks for that document are immediately excluded from other documents' agent searches.
5. **Given** a user views their document list, **When** Private documents are present, **Then** each Private document is visually distinguished (e.g., a badge or label).

---

### Edge Cases

- What happens when a user tries to access a document that doesn't exist (vs. one that exists but belongs to someone else)? Both should return the same response to avoid enumeration.
- What happens to invite codes if the registration process fails partway through (e.g., username already taken)? The code must remain valid so the user can retry.
- What if a user has no documents yet — does the login/registration flow still work?
- What happens to sources and chunks when a document's privacy flag is toggled? The change must take effect immediately for all subsequent agent searches, with no stale data served.
- What if an invite code is never used — does it persist in the system indefinitely?

## Requirements *(mandatory)*

### Functional Requirements

#### Access Control

- **FR-001**: System MUST require a valid, unused invite code to create a new account; accounts cannot be created without one.
- **FR-002**: System MUST mark an invite code as used immediately upon successful account creation, preventing reuse.
- **FR-003**: System MUST authenticate users with an email address and password on every session start.
- **FR-004**: System MUST redirect unauthenticated users to the login page for any protected resource.
- **FR-005**: System MUST maintain a user session so the user does not need to log in on every page load.

#### Data Isolation

- **FR-006**: System MUST scope all document, source, and chunk queries to the authenticated user's identity; no cross-user data may be returned.
- **FR-007**: System MUST return a not-found or forbidden response when a user attempts to access another user's resource by direct identifier.
- **FR-008**: Agents MUST only retrieve sources and chunks belonging to the currently authenticated user.

#### Admin CLI

- **FR-009**: A CLI script MUST allow an administrator to generate one or more invite codes without using the web interface.
- **FR-010**: Each generated invite code MUST be globally unique.
- **FR-011**: The CLI script MUST output codes in plain text to standard output so they can be copied or piped.
- **FR-012**: The CLI script MUST accept an optional count argument to generate multiple codes in a single run.
- **FR-020**: The CLI script MUST provide a password-reset command that allows an administrator to set a new password for any user identified by their email address.

#### Shared Knowledge Within User

- **FR-013**: When an agent performs a knowledge search, it MUST include sources and chunks from all of the user's non-private documents, not just the current document.
- **FR-014**: Sources and chunks added under one document MUST be discoverable by agents operating in any other non-private document of the same user.

#### Document Privacy

- **FR-015**: Each document MUST support a Private flag that the user can toggle at any time.
- **FR-016**: When a document is marked Private, its sources and chunks MUST be excluded from knowledge searches originating from any other document.
- **FR-017**: When a document is marked Private, its sources and chunks MUST still be accessible to agents operating within that same document.
- **FR-018**: Toggling the Private flag on or off MUST take effect immediately for all subsequent searches; no caching may serve stale visibility state.
- **FR-019**: Private documents MUST be visually distinguished in the document list.

### Key Entities

- **User**: A registered person identified by a unique email address. Owns all documents, sources, and chunks. Authenticates with email and password.
- **Invite Code**: A single-use token generated by an administrator. Links to a user account once consumed. No expiry by default.
- **Document**: Belongs to exactly one user. Has a Private flag (default: off). Contains or references sources.
- **Source**: Belongs to exactly one user. Associated with the document it was added from. Carries a privacy context derived from its document's Private flag.
- **Chunk**: A subdivision of a source used for knowledge retrieval. Belongs to exactly one user. Inherits privacy visibility from its source/document.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A user with a valid invite code can complete registration and reach their workspace in under 2 minutes.
- **SC-002**: Zero documents, sources, or chunks belonging to user A are visible to user B under any access path (direct URL, listing, or agent search).
- **SC-003**: An agent in document B successfully retrieves content from a source added to document A (same user, both non-private) within the normal agent response time.
- **SC-004**: After marking a document Private, an agent in another document returns zero results from that document's sources in all subsequent searches.
- **SC-005**: After removing the Private flag from a document, its sources become discoverable by agents in other documents within the same session.
- **SC-006**: An administrator can generate 10 unique invite codes with a single CLI command in under 5 seconds.
- **SC-007**: An invite code used for registration cannot be reused; a second registration attempt with the same code fails with a clear error.
- **SC-008**: All existing features (document editing, source management, agent chat) work correctly for authenticated users with no degradation in experience.

## Clarifications

### Session 2026-03-17

- Q: How should pre-existing data (documents, sources, chunks with no owner) be handled at migration time? → A: Delete all existing data — clean slate, only new user-owned data going forward.
- Q: Should there be a mechanism for users to recover a forgotten password? → A: No self-service recovery — admin resets the password via a CLI command.
- Q: What is the login identifier for users? → A: Email address (unique, stored at registration, used as the login credential and admin-facing identifier).

## Assumptions

- Email address and password is the authentication mechanism; no external identity providers (OAuth, SSO) are in scope for this feature. Email addresses must be unique across all users.
- Sessions are maintained via a secure server-side session cookie; session duration follows standard web application defaults (e.g., expires on browser close or after 24 hours of inactivity).
- Invite codes do not expire; they remain valid until used.
- The CLI script runs on the same server/environment as the application and has direct access to the data store.
- "Private" applies at the document level; there is no per-source privacy toggle.
- When a document is deleted, its sources and chunks are also deleted (existing behaviour assumed to carry over).
- Migration to multi-user performs a clean-slate data deletion; any documents, sources, and chunks created before multi-user support was enabled are purged. No data migration or ownership assignment is performed.
- No admin web UI is in scope; all administrative actions are CLI-only (invite code generation, password reset).
- There is no self-service password recovery; forgotten passwords require an administrator to run the CLI reset command.
- There is no concept of roles beyond "user" and the implied "admin who runs CLI" — all users have identical permissions within their own data.
