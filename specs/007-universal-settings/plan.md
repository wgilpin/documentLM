# Implementation Plan: Universal App Settings

**Branch**: `007-universal-settings` | **Date**: 2026-03-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/007-universal-settings/spec.md`

## Summary

Add a persistent, single-row `user_settings` table and a settings modal (gear icon left of Delete) where the user can set their display name, preferred language, and universal AI instructions. The saved AI instructions, name, and language are injected into the `ChatAgent` system prompt for every session — no per-session setup required.

## Implementation Phases

### Phase 1: Data layer (DB + ORM + schemas + service + tests)

1. **Add `UserSettings` ORM model** to `src/writer/models/db.py`
   - Columns: `id` (int PK=1), `display_name` (str|None), `language_code` (str default "en"), `ai_instructions` (str|None, max 2000), `updated_at`

2. **Add Pydantic schemas** to `src/writer/models/schemas.py`
   - `UserSettingsUpdate`: writable fields + `ai_instructions` max_length=2000
   - `UserSettingsResponse`: read fields (from_attributes=True)

3. **Create `settings_service.py`** with:
   - `async def get_settings(db: AsyncSession) -> UserSettingsResponse` — fetches row id=1 or returns default
   - `async def upsert_settings(db: AsyncSession, data: UserSettingsUpdate) -> UserSettingsResponse` — upsert id=1

4. **Write `tests/unit/test_settings_service.py`** (TDD — write tests first):
   - `test_get_settings_returns_defaults` — no row in DB → defaults returned
   - `test_upsert_settings_creates_row` — upsert on empty DB creates row
   - `test_upsert_settings_updates_existing` — second upsert overwrites
   - `test_ai_instructions_max_length` — Pydantic rejects >2000 chars

5. **Alembic migration** — `add_user_settings` table

### Phase 2: Agent wiring

1. **Modify `make_chat_agent`** to accept `user_settings: UserSettingsResponse | None = None`
   - Build a suffix from settings: name context, language instruction, custom instructions
   - Append suffix to `_INSTRUCTION`

2. **Modify `invoke_chat_agent`** in `chat_service.py` to:
   - Accept `user_settings: UserSettingsResponse | None = None`
   - Pass it through to `make_chat_agent`

3. **Modify `process_chat`** in `chat_service.py` to:
   - Call `settings_service.get_settings(db)` before invoking agent
   - Pass result to `invoke_chat_agent`

### Phase 3: API endpoint

1. **Create `src/writer/api/settings.py`**:
   - `GET /api/settings` → `UserSettingsResponse` JSON
   - `POST /api/settings` → upsert, return `UserSettingsResponse` + `HX-Trigger: settingsSaved` header

2. **Register router** in `src/writer/main.py`

### Phase 4: UI

1. **Add settings modal to `document.html`**:
   - Gear icon (`⚙`) button immediately left of the Delete button
   - `<dialog id="settings-dialog">` with form:
     - `display_name` text input
     - `language_code` select (11 options from spec)
     - `ai_instructions` textarea (maxlength=2000 + inline character counter)
     - Save / Cancel buttons
   - HTMX form post to `POST /api/settings`
   - `HX-Trigger: settingsSaved` → close dialog via minimal inline JS

2. **Pre-populate modal** by loading settings via `GET /api/settings` when the modal opens (HTMX `hx-get` on dialog open, or pass settings in Jinja2 template context)

## Complexity Tracking

No constitution violations. Single-row upsert is the simplest pattern for single-user settings storage.
